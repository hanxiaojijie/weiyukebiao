## 二、测试目标

本次测试的目标，是验证以下核心闭环是否成立：

```text
用户进入 -> 查看课程 -> 新增排期 -> 当天打卡 -> 填写学习记录 -> 学分变化 -> 奖励进度变化 -> 统计页同步
```

如果这条链路能稳定跑通，说明当前版本的主功能可用。

## 三、测试范围

### 本次在测范围

- 用户初始化与数据隔离
- 首页总览
- 课程录入 / 编辑 / 详情
- 个人课表编排 / 编辑 / 停用
- 打卡状态流转
- 学习记录保存
- 学分规则维护
- 奖励目标维护
- 统计中心
- 主题切换

### 本次不作为重点验收范围

- AI 能力
- 语音整理
- 海报生成
- 真实文件上传
- 提醒通知
- 高并发压测

## 七、执行顺序建议

请测试人员按下面顺序执行，不建议一上来随机点页面。

### 第一轮：主链路冒烟

1. 账号 A 首次进入
2. 新增课程
3. 新增排期
4. 进入打卡页执行上课打卡
5. 执行下课打卡
6. 保存学习记录
7. 查看首页
8. 查看奖励页
9. 查看统计页

### 第二轮：边界与异常

1. 课程必填校验
2. 排期冲突校验
3. 请假原因校验
4. 休息次数上限校验
5. 非法时间范围校验

### 第三轮：隔离与回归

1. 切换账号 B 登录
2. 验证账号 A 的个人数据不可见
3. 验证共享课程库仍可见
4. 再切回账号 A，验证原数据未丢失

## 八、重点关注的验证点

### 1. 数据一致性

重点核对：

- 首页学分与统计页学分是否一致
- 奖励页进度与首页奖励进度是否一致
- 打卡完成后首页和统计是否同步更新

### 2. 状态流转

重点核对：

- 待开始 -> 进行中 -> 已完成 是否符合预期
- 已请假 / 已休息 / 已旷课状态是否只出现在正确场景
- 已完成后是否还能修改学习记录

### 3. 用户隔离

重点核对：

- 用户 A 的排期、打卡、规则、奖励，用户 B 不应看到
- 共享课程库可被多个用户查看

### 4. 课表正确性

重点核对：

- 新增排期后课表是否立即可见
- 编辑排期后展示是否同步更新
- 停用排期后是否从课表中移除
- 回到今天和左右滑动是否正常

## 九、通过标准

当前版本可视为“通过测试”的最低标准：

- 主链路无阻断问题
- 数据不会串用户
- 打卡结果能正确同步到首页、奖励页、统计页
- 排期与课表展示一致
- 关键校验项有效
